# gRPC-OpenHarmony 架构设计文档

## 1. 项目概述

### 1.1 背景
- **鸿蒙Next环境**：已有Protobuf序列化库，但缺乏完整的gRPC实现
- **gRPC核心**：基于HTTP/2的RPC框架，支持多种调用模式（Unary、Server Streaming、Client Streaming、Bidirectional Streaming）
- **目标**：在鸿蒙Next上实现一个高性能、易用的gRPC三方库

### 1.2 技术方案
**采用NAPI（Native API）方案**：基于gRPC官方C++ Core，通过NAPI绑定层提供ArkTS接口

### 1.3 核心技术难点
1. gRPC C++库在鸿蒙平台的交叉编译和适配
2. NAPI跨语言类型转换和内存管理
3. 异步调用的Promise/AsyncIterator适配
4. ArkTS Protobuf库与Native层的字节流传递
5. 线程安全的回调机制（ThreadSafeFunction）

│  └──────────────────────────────────────────────┘  │
├─────────────────────────────────────────────────────┤
│         NAPI绑定层（C++ Binding）                     │

│  │  gRPC C++ Core (grpc++)                       │  │
│  │  - Channel管理 │ Call执行                      │  │
│  │  - HTTP/2传输 │ 流控 │ 多路复用               │  │

**架构特点**：
- 🎯 **职责清晰**：ArkTS层负责API设计和序列化，Native层负责传输
- ⚡ **性能优异**：核心逻辑在Native层执行，字节流零拷贝传递
- 🔧 **易于维护**：复用gRPC C++ Core，减少80%+开发维护工作
- 🔄 **灵活可扩展**：上层逻辑可灵活调整，底层稳定可靠

## 3. 核心模块设计

### 3.1 传输层（Transport Layer）

#### 3.1.1 NAPI方案传输层架构

**采用gRPC C++ Core内置的HTTP/2实现**：
- ✅ **无需自研**：直接使用gRPC官方已实现并充分测试的HTTP/2协议栈
- ✅ **功能完整**：支持完整的HTTP/2特性（多路复用、流控、HPACK压缩等）
- ✅ **性能优异**：C++实现，经过大量优化
- ✅ **持续更新**：跟随gRPC官方版本获得最新特性和安全补丁

**Native层传输实现**：
```cpp
// gRPC C++ Core内部已实现
// 我们只需通过NAPI绑定暴露接口
class TransportWrapper {
  // gRPC内部自动处理：
  // - HTTP/2连接管理
  // - 流多路复用
  // - 流量控制
  // - HPACK头部压缩
  // - 连接保活（PING/PONG）
  
  // NAPI层只需要：
  // 1. 创建Channel（grpc::CreateChannel）
  // 2. 传递字节流（grpc::ByteBuffer）
  // 3. 处理异步回调
};
```

#### 3.1.2 网络层对接
gRPC C++ Core会自动使用系统底层的Socket API，在鸿蒙上通过NDK提供的POSIX兼容接口连接：

```cpp
// gRPC内部使用标准POSIX Socket
// 鸿蒙NDK提供POSIX兼容层：
// - socket() / connect() / send() / recv()
// - OpenSSL for TLS (或鸿蒙安全组件)

// NAPI层无需关心底层网络细节
class ChannelWrapper {
  static napi_value CreateChannel(napi_env env, const char* target) {
    // gRPC自动处理网络连接
    auto channel = grpc::CreateChannel(
      target,
      grpc::InsecureChannelCredentials()  // 或 SslCredentials
    );
    return WrapChannel(env, channel);
  }
};
```

**关键优势**：
- 无需实现网络适配层，gRPC内部已处理
- 自动利用鸿蒙系统的网络能力
- 完整的TLS/SSL支持（基于OpenSSL或鸿蒙安全组件）

interface LoadBalancer {
  // 选择一个可用的Subchannel
  pick(): Subchannel;
  // 更新后端地址列表
  updateAddresses(addresses: Address[]): void;
}

// 内置策略
class RoundRobinLoadBalancer implements LoadBalancer {
  private subchannels: Subchannel[];
  private currentIndex: number = 0;
  
  pick(): Subchannel {
    const subchannel = this.subchannels[this.currentIndex];
    this.currentIndex = (this.currentIndex + 1) % this.subchannels.length;
    return subchannel;
  }
}

### 3.4 Server端设计

```typescript
class Server {
  private services: Map<string, ServiceDefinition>;
  private handlers: Map<string, Handler>;
  private listener: NetworkListener;
  
  // 注册服务
  addService(definition: ServiceDefinition, implementation: any): void {
    for (const [methodName, methodDef] of Object.entries(definition)) {
      const fullMethodName = `/${definition.serviceName}/${methodName}`;
      this.handlers.set(fullMethodName, {
        func: implementation[methodName],
        requestType: methodDef.requestType,
        responseType: methodDef.responseType,
        streamType: methodDef.streamType
      });
    }
  }
  
  // 启动服务器
  async start(): Promise<void> {
    this.listener.on('request', async (stream: Http2Stream) => {
      const method = stream.headers[':path'];
      const handler = this.handlers.get(method);
      
      if (!handler) {
        stream.sendHeaders({ ':status': '404' });
        return;
      }
      
      await this.handleRequest(stream, handler);
    });
  }
  
  private async handleRequest(stream: Http2Stream, handler: Handler) {
    // 根据streamType处理不同类型的调用
    switch (handler.streamType) {
      case 'unary':
        await this.handleUnary(stream, handler);
        break;
      case 'serverStream':
        await this.handleServerStream(stream, handler);
        break;
      // ...
    }
  }
}
```

### 6.2 NAPI优势分析

#### ✅ 核心优势

1. **复用成熟代码**
   - 直接使用gRPC官方C++ Core（grpc++），经过充分测试
   - 完整支持所有gRPC特性（xDS、Channelz、健康检查等）
   - 省去HTTP/2协议栈开发（80%+工作量）

2. **性能最优**
   - Native代码执行效率高
   - 零拷贝：Protobuf序列化在Native层完成
   - 利用C++的并发优势（线程池、异步IO）

3. **开发周期短**
   - 核心逻辑复用，只需编写NAPI绑定
   - 预计开发周期：4-6周（vs 纯TS 10-12周）

4. **长期可维护**
   - 跟随gRPC官方更新，及时获得新特性和安全补丁
   - 社区支持强大

#### ⚠️ 挑战和风险

1. **NAPI复杂度**
   - 需要处理跨语言类型转换（napi_value ↔ C++对象）
   - 异步回调适配（Promise、AsyncIterator）
   - 内存管理：防止内存泄漏和野指针

2. **调试困难**
   - 需要同时调试ArkTS和C++代码
   - 崩溃定位困难（可能在Native层）

3. **平台兼容性**
   - 依赖鸿蒙NDK，API可能随版本变化
   - 需要为不同架构编译（ARM、ARM64、x86_64）

4. **包体积增大**
   - Native库大小约2-5MB（取决于裁剪程度）

### 6.3 Protobuf与NAPI联动方案

#### 6.3.1 核心设计理念

**关键决策：字节流传递模式（Byte Stream Pass-Through）**

```
┌─────────────────────────────────────────────────────┐
│  ArkTS层                                             │
│  ┌───────────────────────────────────────────────┐ │
│  │ 用户代码                                        │ │
│  │ request = new HelloRequest({ name: 'World' })  │ │
│  └───────────────┬─────────────────────────────────┘ │
│                  ↓                                    │
│  ┌───────────────────────────────────────────────┐  │
│  │ ArkTS Protobuf库（已有）                        │  │
│  │ requestBytes = request.serializeBinary()       │  │
│  │ // 返回 Uint8Array                             │  │
│  └───────────────┬─────────────────────────────────┘ │
└──────────────────┼───────────────────────────────────┘
                   ↓ 字节数组传递
┌──────────────────┼───────────────────────────────────┐
│  NAPI绑定层      ↓                                    │
│  ┌───────────────────────────────────────────────┐  │
│  │ 类型转换：napi_value → std::vector<uint8_t>   │  │
│  │ 无需理解Protobuf格式                           │  │
│  └───────────────┬─────────────────────────────────┘ │
└──────────────────┼───────────────────────────────────┘
                   ↓ 原始字节
┌──────────────────┼───────────────────────────────────┐
│  gRPC Native层   ↓                                    │
│  ┌───────────────────────────────────────────────┐  │
│  │ gRPC发送原始字节流（不解析内容）               │  │
│  │ grpc::ByteBuffer 或 std::string                │  │
│  └───────────────┬─────────────────────────────────┘ │
│                  ↓                                    │
│              网络传输                                 │
│                  ↓                                    │
│  ┌───────────────────────────────────────────────┐  │
│  │ 接收响应字节流                                  │  │
│  └───────────────┬─────────────────────────────────┘ │
└──────────────────┼───────────────────────────────────┘
                   ↓ 原始字节
┌──────────────────┼───────────────────────────────────┐
│  NAPI绑定层      ↓                                    │
│  ┌───────────────────────────────────────────────┐  │
│  │ 类型转换：std::vector<uint8_t> → napi_value   │  │
│  │ 返回 Uint8Array 到ArkTS                        │  │
│  └───────────────┬─────────────────────────────────┘ │
└──────────────────┼───────────────────────────────────┘
                   ↓ 字节数组
┌──────────────────┼───────────────────────────────────┐
│  ArkTS层         ↓                                    │
│  ┌───────────────────────────────────────────────┐  │
│  │ ArkTS Protobuf库反序列化                        │  │
│  │ response = HelloResponse.deserializeBinary(    │  │
│  │   responseBytes                                │  │
│  │ )                                              │  │
│  └───────────────┬─────────────────────────────────┘ │
│                  ↓                                    │
│  ┌───────────────────────────────────────────────┐  │
│  │ 用户代码                                        │  │
│  │ console.log(response.message)                  │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

**设计优势**：
1. ✅ **充分复用**：使用鸿蒙已有的ArkTS Protobuf库
2. ✅ **架构清晰**：NAPI层只处理字节流，职责单一
3. ✅ **无重复依赖**：无需在Native层引入Protobuf C++库
4. ✅ **灵活性高**：与任何序列化格式兼容（JSON、MessagePack等）
5. ✅ **调试友好**：序列化层在ArkTS，易于调试

#### 6.3.2 详细实现方案

##### 方案一：完全字节流模式（推荐）

**Native层实现（Generic Call）**：
```cpp
// native/src/call_wrapper.cpp

#include <napi/native_api.h>
#include <grpcpp/grpcpp.h>
#include <vector>

// 通用的gRPC调用包装器 - 不依赖具体的Protobuf消息类型
class GenericCallWrapper {
public:
    // Unary调用：接收字节流，返回字节流
    static napi_value MakeUnaryCall(napi_env env, napi_callback_info info) {
        size_t argc = 4;
        napi_value args[4]; // [channelHandle, method, requestBytes, options]
        napi_get_cb_info(env, info, &argc, args, nullptr, nullptr);
        
        // 1. 提取参数
        void* channel_ptr;
        napi_unwrap(env, args[0], &channel_ptr);
        auto channel = static_cast<std::shared_ptr<grpc::Channel>*>(channel_ptr);
        
        // 2. 获取方法名
        char method[256];
        size_t method_len;
        napi_get_value_string_utf8(env, args[1], method, sizeof(method), &method_len);
        
        // 3. 提取请求字节流 (Uint8Array → std::vector<uint8_t>)
        std::vector<uint8_t> request_bytes;
        if (!ExtractBytes(env, args[2], request_bytes)) {
            napi_throw_error(env, nullptr, "Invalid request bytes");
            return nullptr;
        }
        
        // 4. 创建Promise
        napi_value promise;
        napi_deferred deferred;
        napi_create_promise(env, &deferred, &promise);
        
        // 5. 准备异步调用数据
        auto* call_data = new AsyncUnaryCallData{
            .env = env,
            .deferred = deferred,
            .channel = *channel,
            .method = std::string(method),
            .request_bytes = std::move(request_bytes),
            .response_bytes = {},
            .status = {}
        };
        
        // 6. 创建异步工作
        napi_value resource_name;
        napi_create_string_utf8(env, "grpc_unary_call", NAPI_AUTO_LENGTH, &resource_name);
        
        napi_create_async_work(
            env, nullptr, resource_name,
            ExecuteUnaryCall,  // 后台线程执行
            CompleteUnaryCall, // 主线程回调
            call_data,
            &call_data->work
        );
        
        napi_queue_async_work(env, call_data->work);
        
        return promise;
    }
    
private:
    struct AsyncUnaryCallData {
        napi_env env;
        napi_deferred deferred;
        napi_async_work work;
        std::shared_ptr<grpc::Channel> channel;
        std::string method;
        std::vector<uint8_t> request_bytes;
        std::vector<uint8_t> response_bytes;
        grpc::Status status;
    };
    
    // 后台线程执行gRPC调用
    static void ExecuteUnaryCall(napi_env env, void* data) {
        auto* call_data = static_cast<AsyncUnaryCallData*>(data);
        
        // 使用gRPC Generic API（不需要知道消息类型）
        grpc::ClientContext context;
        
        // 创建Generic Stub（通用存根，接受任意消息）
        grpc::GenericStub stub(call_data->channel);
        
        // 准备请求（原始字节）
        grpc::ByteBuffer request_buffer;
        grpc::Slice request_slice(
            call_data->request_bytes.data(),
            call_data->request_bytes.size()
        );
        request_buffer.Clear();
        request_buffer.TrySingleSlice(&request_slice);
        
        // 执行调用
        grpc::ByteBuffer response_buffer;
        call_data->status = stub.UnaryCall(
            &context,
            call_data->method,
            request_buffer,
            &response_buffer
        );
        
        // 提取响应字节
        if (call_data->status.ok()) {
            std::vector<grpc::Slice> slices;
            response_buffer.Dump(&slices);
            
            for (const auto& slice : slices) {
                call_data->response_bytes.insert(
                    call_data->response_bytes.end(),
                    slice.begin(),
                    slice.end()
                );
            }
        }
    }
    
    // 主线程完成回调
    static void CompleteUnaryCall(napi_env env, napi_status status, void* data) {
        auto* call_data = static_cast<AsyncUnaryCallData*>(data);
        
        if (call_data->status.ok()) {
            // 成功：返回响应字节流
            napi_value result = CreateUint8Array(env, call_data->response_bytes);
            napi_resolve_deferred(env, call_data->deferred, result);
        } else {
            // 失败：创建错误对象
            napi_value error = CreateGrpcError(
                env,
                call_data->status.error_code(),
                call_data->status.error_message()
            );
            napi_reject_deferred(env, call_data->deferred, error);
        }
        
        // 清理资源
        napi_delete_async_work(env, call_data->work);
        delete call_data;
    }
    
    // 工具函数：从napi_value提取字节数组
    static bool ExtractBytes(napi_env env, napi_value value, std::vector<uint8_t>& out) {
        bool is_typedarray;
        napi_is_typedarray(env, value, &is_typedarray);
        if (!is_typedarray) return false;
        
        napi_typedarray_type type;
        size_t length;
        void* data;
        napi_value arraybuffer;
        size_t byte_offset;
        
        napi_get_typedarray_info(
            env, value, &type, &length, &data, &arraybuffer, &byte_offset
        );
        
        if (type != napi_uint8_array) return false;
        
        uint8_t* bytes = static_cast<uint8_t*>(data);
        out.assign(bytes, bytes + length);
        
        return true;
    }
    
    // 工具函数：创建Uint8Array
    static napi_value CreateUint8Array(napi_env env, const std::vector<uint8_t>& bytes) {
        napi_value arraybuffer;
        void* data;
        
        napi_create_arraybuffer(env, bytes.size(), &data, &arraybuffer);
        std::memcpy(data, bytes.data(), bytes.size());
        
        napi_value uint8array;
        napi_create_typedarray(
            env, napi_uint8_array, bytes.size(), arraybuffer, 0, &uint8array
        );
        
        return uint8array;
    }
    
    // 工具函数：创建gRPC错误对象
    static napi_value CreateGrpcError(
        napi_env env,
        int code,
        const std::string& message
    ) {
        napi_value error;
        napi_value js_message;
        napi_create_string_utf8(env, message.c_str(), NAPI_AUTO_LENGTH, &js_message);
        napi_create_error(env, nullptr, js_message, &error);
        
        // 添加gRPC状态码
        napi_value js_code;
        napi_create_int32(env, code, &js_code);
        napi_set_named_property(env, error, "code", js_code);
        
        return error;
    }
};
```

**ArkTS封装层实现**：
```typescript
// src/client.ts

import grpcNative from 'libgrpc_openharmony.so';

/**
 * gRPC Channel - 管理连接
 */
export class Channel {
  private nativeHandle: any;
  
  constructor(target: string, credentials?: ChannelCredentials, options?: ChannelOptions) {
    this.nativeHandle = grpcNative.createChannel(
      target,
      credentials?.toNative() || null,
      options || {}
    );
  }
  
  /**
   * 创建调用对象
   */
  createCall(method: string, deadline?: Date): Call {
    return new Call(this.nativeHandle, method, deadline);
  }
  
  close(): void {
    grpcNative.closeChannel(this.nativeHandle);
  }
}

/**
 * gRPC Call - 封装单次RPC调用
 */
export class Call {
  constructor(
    private channelHandle: any,
    private method: string,
    private deadline?: Date
  ) {}
  
  /**
   * Unary调用 - 核心：字节流传递
   * @param requestBytes 序列化后的请求字节（由Protobuf库生成）
   * @returns 响应字节（需要用Protobuf库反序列化）
   */
  async unaryCallBytes(requestBytes: Uint8Array): Promise<Uint8Array> {
    const options = this.deadline ? { deadline: this.deadline.getTime() } : {};
    
    // 调用Native层：传递字节流，返回字节流
    const responseBytes = await grpcNative.makeUnaryCall(
      this.channelHandle,
      this.method,
      requestBytes,
      options
    );
    
    return responseBytes;
  }
}

/**
 * 类型化的Client封装 - 提供类型安全的API
 */
export class TypedClient<TService> {
  constructor(
    private channel: Channel,
    private serviceName: string
  ) {}
  
  /**
   * 创建类型化的Unary方法调用器
   */
  makeUnaryMethod<TRequest, TResponse>(
    methodName: string,
    requestSerializer: (req: TRequest) => Uint8Array,
    responseDeserializer: (bytes: Uint8Array) => TResponse
  ): (request: TRequest, options?: CallOptions) => Promise<TResponse> {
    
    return async (request: TRequest, options?: CallOptions): Promise<TResponse> => {
      const fullMethod = `/${this.serviceName}/${methodName}`;
      const call = this.channel.createCall(fullMethod, options?.deadline);
      
      // 1. 序列化请求（使用ArkTS Protobuf库）
      const requestBytes = requestSerializer(request);
      
      // 2. 发送到Native层，获取响应字节
      const responseBytes = await call.unaryCallBytes(requestBytes);
      
      // 3. 反序列化响应（使用ArkTS Protobuf库）
      const response = responseDeserializer(responseBytes);
      
      return response;
    };
  }
  
  /**
   * 创建Server Streaming方法调用器
   */
  makeServerStreamMethod<TRequest, TResponse>(
    methodName: string,
    requestSerializer: (req: TRequest) => Uint8Array,
    responseDeserializer: (bytes: Uint8Array) => TResponse
  ): (request: TRequest, options?: CallOptions) => AsyncIterableIterator<TResponse> {
    
    return async function*(request: TRequest, options?: CallOptions): AsyncIterableIterator<TResponse> {
      const fullMethod = `/${this.serviceName}/${methodName}`;
      const requestBytes = requestSerializer(request);
      
      // 获取Native层的字节流迭代器
      const nativeIterator = grpcNative.makeServerStreamCall(
        this.channel['nativeHandle'],
        fullMethod,
        requestBytes,
        options || {}
      );
      
      // 迭代并反序列化每个响应
      for await (const responseBytes of nativeIterator) {
        yield responseDeserializer(responseBytes);
      }
    }.bind(this);
  }
}

// 调用选项
export interface CallOptions {
  deadline?: Date;
  metadata?: Metadata;
  credentials?: CallCredentials;
}

    }
}
```

#### 6.7.3 错误处理
```cpp
napi_value CreateGrpcError(napi_env env, const grpc::Status& status) {
    napi_value error;
    napi_create_error(env, nullptr, 
        napi_create_string_utf8(env, status.error_message().c_str(), ...),
        &error);
    
    // 添加gRPC特有属性
    napi_value code;
    napi_create_int32(env, status.error_code(), &code);
    napi_set_named_property(env, error, "code", code);
    
    return error;
}
```

## 8. 性能优化考虑

### 7.1 内存优化
- **对象池**：复用Frame、Buffer对象
- **零拷贝**：使用Uint8Array直接传递数据
- **及时释放**：Stream关闭后清理资源

### 7.2 网络优化
- **连接复用**：HTTP/2多路复用
- **流控**：合理的窗口大小
- **压缩**：HPACK头部压缩，可选消息压缩（gzip）

### 7.3 并发优化
- **异步非阻塞**：充分利用Promise/async
- **TaskPool**：CPU密集任务（序列化）分离
- **批量操作**：合并小消息

### D. 术语表

- **Channel**：客户端到服务器的逻辑连接
- **Subchannel**：实际的物理TCP连接
- **Call**：单次RPC调用
- **Stream**：HTTP/2流
- **Frame**：HTTP/2帧
- **Metadata**：RPC元数据（类似HTTP头）
- **Stub**：客户端代理或服务器骨架代码
- **NAPI (Native API)**：鸿蒙提供的C/C++本地接口，用于编写高性能本地模块
- **NDK (Native Development Kit)**：鸿蒙原生开发套件
- **ArkTS**：鸿蒙的增强型TypeScript语言
- **ThreadSafeFunction**：NAPI提供的线程安全回调机制
- **napi_value**：NAPI中JavaScript值的C表示
- **Binding**：将本地代码暴露给JavaScript的绑定层
